# Счетчик очков (Score Counter)

Простое приложение-счетчик для настольных игр, созданное в учебных целях для изучения основ Android-разработки на Kotlin с использованием Jetpack Compose.

## Описание игры
Приложение представляет собой экран с отображением текущего счета и двумя кнопками: увеличения и уменьшения значения. Оно демонстрирует реактивный подход к управлению состоянием интерфейса.

## Технологии
Проект построен на современном стеке технологий:
*   **Kotlin** — основной язык разработки.
*   **Jetpack Compose** — инструмент для создания пользовательского интерфейса (декларативный подход).
*   **ViewModel** — компонент архитектуры для хранения и управления UI-данными, устойчивый к изменению конфигурации.
*   **StateFlow / MutableStateFlow** — компонент из Kotlin Coroutines для хранения состояния и реактивной передачи данных UI.

## Функциональность
Приложение обладает следующим базовым функционалом:
*   Отображение текущего значения счетчика.
*   Увеличение значения счетчика на 1 при нажатии кнопки "Плюс".
*   Уменьшение значения счетчика на 1 при нажатии кнопки "Минус".
*   Сохранение состояния счетчика при повороте экрана (благодаря ViewModel).
* 1. Что такое ViewModel и зачем он нужен?
     Объяснение: ViewModel — это специальный класс, который хранит данные, необходимые для интерфейса (UI), и управляет ими. Его главная задача — переживать повороты экрана и другие изменения конфигурации (например, смену языка или темной темы). Он решает проблему потери данных при пересоздании Activity/Fragment.

Что происходит при повороте экрана без ViewModel? Без ViewModel данные обычно хранятся в самой Activity. При повороте экрана Activity уничтожается и создается заново. Все переменные, которые были в ней, сбрасываются до значений по умолчанию. Например, наш счетчик обнулился бы.

2. В чём разница между MutableStateFlow и StateFlow?
   Разница: MutableStateFlow (изменяемый) позволяет записывать в него новые значения (у него есть метод .value = ...). StateFlow (неизменяемый) позволяет только читать текущее значение и подписываться на его обновления, но не изменять его извне.

Зачем два поля (_uiState и uiState)? Это реализация паттерна "бэкинг проперти" (см. ниже). Публичное поле uiState: StateFlow открыто для чтения всему миру (в том числе UI), а приватное поле _uiState: MutableStateFlow доступно для изменения только внутри самой ViewModel. Это обеспечивает инкапсуляцию — никто случайно не испортит данные извне.

Что такое “backing property pattern”? Это шаблон проектирования в Kotlin, когда для одного свойства объявляется два поля: приватное (для изменения) и публичное (для чтения), которое возвращает приватное в неизменяемом виде.

3. Почему в ViewModel нельзя хранить Context или View?
   Причина: ViewModel живет дольше, чем Activity/View. Если сохранить ссылку на Activity (контекст) в ViewModel, а Activity захочет удалиться ("умереть") при повороте экрана, Garbage Collector (сборщик мусора) не сможет этого сделать, потому что на него всё ещё ссылается ViewModel.

Что произойдет? Это приведет к утечке памяти (memory leak). Приложение будет держать в памяти кучу мертвых экранов, что рано или поздно вызовет падение приложения из-за нехватки памяти (OutOfMemoryError).

4. Чем StateFlow отличается от remember и rememberSaveable?
   remember: Хранит состояние только внутри Composition (внутри одного экрана/функции). При повороте экрана состояние сбрасывается.

rememberSaveable: Хранит состояние внутри Composition, но сохраняет его в Bundle. При повороте экрана состояние восстанавливается.

StateFlow: Хранит состояние в ViewModel (вне Composition). При повороте экрана состояние сохраняется, так как ViewModel переиспользуется.

Когда что использовать: remember — для временных состояний внутри UI (например, развернут ли спойлер). rememberSaveable — когда лень заводить ViewModel ради одного примитива, но данные нужно сохранять при повороте. StateFlow — для сложной бизнес-логики и данных, которые должны жить весь lifecycle экрана.

5. Что такое data class в Kotlin и зачем он используется для UI State?
   Data Class — это класс, предназначенный исключительно для хранения данных.

Зачем для UI State: Удобно хранить состояние экрана в одном объекте (например, data class CounterState(val count: Int, val isLoading: Boolean)). Это делает код предсказуемым.

Какие методы генерируются? Автоматически генерируются: toString(), equals(), hashCode(), и copy().

Зачем copy(): Data Class по умолчанию неизменяемый (val). Чтобы создать новый объект на основе старого с изменениями, используется функция copy(). Пример: _uiState.update { it.copy(count = it.count + 1) }.

6. Как работает collectAsState() в Compose?
   collectAsState() — это функция-расширение Compose, которая "слушает" поток StateFlow. Когда в StateFlow появляется новое значение, эта функция преобразует его в State<T> (обычный State из Compose). Как только State меняется, Compose автоматически перерисовывает (рекомпозирует) все места на экране, где значение этого State используется.

7. Зачем разделять код на пакеты data и ui?
   Разделение на пакеты (слои) нужно для соблюдения принципов чистой архитектуры.

Принципы: Это поддерживает Single Responsibility Principle (каждый слой отвечает за свое) и Dependency Rule (зависимости направлены внутрь, UI зависит от data, но не наоборот). Это упрощает тестирование, масштабирование и переиспользование кода.

8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
   Что происходит: Когда ОС убивает процесс приложения (из-за нехватки памяти или пользователь свайпнул приложение из списка недавних), ViewModel уничтожается безвозвратно. Вызывается метод onCleared().

Сохраняются ли данные? Нет, данные в ViewModel не сохраняются. Это оперативная память (RAM).

Какие альтернативы есть для долговременного хранения? Для данных, которые должны жить вечно (даже после закрытия приложения), используются:

DataStore / SharedPreferences (для простых настроек и ключей).

Room Database (SQLite) (для структурированных данных, списков задач и т.д.).

SavedStateHandle (особый механизм в ViewModel, который выживает при убийстве процесса, но не при переустановке приложения, работает через Bundle).
